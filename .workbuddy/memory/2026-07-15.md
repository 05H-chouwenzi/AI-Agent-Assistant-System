## AI助手响应延迟深度分析

对比了参考项目 ai-qa-community (GitHub) 与本项目的架构差异，完成了完整的性能瓶颈诊断。

### 发现的关键问题

**P0 严重问题**:
1. `chat_stream.py:99` 双重图执行 BUG — `if not full_answer:` 在 async for 循环内部，导致整图被重复执行，LLM调用翻倍
2. Worker Agent 使用 `streaming=False` — SSE流式名存实亡，用户等完整响应后才收到内容

**P1 显著问题**:
3. FAISS每次搜索从磁盘重载索引和文档 (~100-500ms)
4. RAG Embedding使用同步OpenAI客户端，阻塞async事件循环
5. MySQL工具每次新建连接，无连接池
6. WeatherTool/HttpTool使用同步httpx/urllib
7. ToolRouter每次调用新建OpenAI客户端

**核心架构差异**:
- 参考项目：单次LLM流式调用(stream=True) → SSE逐token推送 → 1秒内响应
- 本项目：LangGraph(supervisor+create_react_agent) → 多次LLM调用 + BUG双重执行 → ~10秒

## 深度对比：本项目 vs WL7749/ai-agent（1秒响应）

获取了参考项目完整源码，逐文件对比。

### 新发现的关键差异：

1. **WebSocket 假流式（之前未发现）**：ws_chat.py:104 用 `ainvoke()` 等完整回答(3-8s)，拿到后再伪流式(`asyncio.sleep(0.01)`)，用户先等8秒空白再看到快速吐字。参考项目用 `astream_events` 真流式，首token 500ms-1s。

2. **SSE 双重执行重新确认**：chat_stream.py:99 的 `if not full_answer` 在 async for 内部并非在内部，而是在循环结束后（第99行确实在async for之后），但仍有 fallback ainvoke — 当流式事件未产出token时触发第二次LLM调用。

3. **同步阻塞工具是实际最大瓶颈**：http_tool(urllib同步)、weather_tool(sync httpx)、mysql_tool(pymysql每次新建连接)、embedding(sync OpenAI) — 全部阻塞异步事件循环。

4. **ToolRouter冗余LLM调用**：tool_router.py:282 每次新建同步OpenAI客户端，292行同步HTTP调用(2-5s)，而LangGraph create_react_agent本身就有function calling能力。

5. **参考项目厉害之处**：
   - 所有工具 `@tool async def` + `httpx.AsyncClient`
   - PostgreSQL + asyncpg（真异步DB）
   - MVP伪embedding（SHA256 hash，0延迟）
   - `astream_events` + `on_chat_model_stream` 真流式
   - 无ToolRouter，无冗余LLM调用

### 新的修复优先级：
- P0: ws_chat.py 改 astream_events 真流式（-5~7s首token延迟）
- P0: chat_stream.py 去fallback双重执行
- P1: 工具全异步化 → 消除事件循环阻塞
- P2: 消除 ToolRouter 冗余 LLM 调用

### 产出
- 生成 `性能对比分析报告.md` — 完整对比分析，含具体行号、时间线分析、修复代码

## 2026-07-15 性能修复实施（保持架构不变）

### 修改了 11 个文件，未改变 Supervisor→Worker 协作流程图：

1. **api/ws_chat.py** — WebSocket 真流式：`ainvoke()` → `astream_events(v2)`，添加 `on_chat_model_stream`/`on_tool_start`/`on_tool_end` 事件，添加 supervisor 路由事件推送，fallback 只在流式完全失败时触发
2. **api/chat_stream.py** — 修复 SSE 双重执行Bug：`not full_answer` → `not graph_events_received`，只有零事件才fallback
3. **api/chat.py** — FastRouter 路径：`execute()` → `await aexecute()` 异步工具调用
4. **tools/langchain_tools.py** — `_arun` 真正异步：优先用 `manager.aexecute()`，兜底 `asyncio.to_thread()`
5. **tools/tool_manager.py** — 新增 `aexecute()` 异步执行方法，优先用工具的 `aexecute`，兜底线程池
6. **tools/weather_tool.py** — 新增 `aexecute()` 用 `httpx.AsyncClient`，抽取 `_parse_weather()` 公共函数
7. **tools/http_tool.py** — 新增 `aexecute()` 用 `httpx.AsyncClient` 替代 urllib
8. **tools/mysql_tool.py** — `execute()` 改用 SQLAlchemy 引擎连接池（`_db_engine.connect()`），`aexecute()` 用 `asyncio.to_thread()` 包装 pymysql
9. **tools/rag_tool.py** — 新增 `aexecute()` 调用 `aretrieve` 异步检索
10. **rag/embedding/__init__.py** — 新增 `AsyncOpenAI` 客户端 + `aembed_text()` 异步向量化
11. **rag/retriever/__init__.py** — 新增 `aretrieve()` 异步检索

### 预期效果：
- WebSocket 首token延迟：8-10s → 1-2s（-80%）
- SSE 无双重执行风险
- 工具调用不再阻塞事件循环（全部通过 asyncio.to_thread 或原生异步）

## 后端启动卡顿问题排查与修复

### 问题
- 用户第二天打开项目，VSCode 启动后端后聊天列表和 AI 对话不可用
- 端口 8000 未监听，但前端 5173 正常运行，MySQL 3306 正常

### 根因
1. **导入时间过长（15.77s）**：
   - `rag/embedding/__init__.py`：模块导入时创建 `OpenAI()` + `AsyncOpenAI()` 客户端对象，各耗时 ~4s（共 9.45s）
   - `agent/workflow/graph.py`：导入时调用 `_build_graph()` 编译 LangGraph（已通过 _GraphProxy 延迟）
   - `agent/graph/state.py`：`from langgraph.graph.message import add_messages` 需 6s（LangGraph 库本身，无法绕开）
2. **reload=True 导致启动时间翻倍**：uvicorn fork 子进程重新导入，总启动 ~30s
3. **reload 残留进程占端口**：VSCode 关闭后子进程可能存活，导致下次启动端口冲突

### 修复（3 个文件）
1. `rag/embedding/__init__.py`：OpenAI/AsyncOpenAI 客户端改为 `@lru_cache` 懒加载
2. `agent/workflow/graph.py`：`agent_graph` 改为 `_GraphProxy` 代理对象，首次访问时才编译
3. `agent/graph/nodes.py`：`from langgraph.prebuilt import create_react_agent` 移入 `_build_agents()` 内部

### 效果
- 导入时间：15.77s → 8.59s（-45%）
- 剩余 8.59s 主要来自 LangGraph 库本身的导入（~6s），无法进一步优化

## 服务器部署 AI 助手崩溃修复

### 问题
- Docker 部署到云服务器后，后端容器崩溃：`ModuleNotFoundError: No module named 'langchain'`
- 服务器 `.env` 缺少 `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`

### 根因
1. `requirements.txt` 有 `langchain-core`、`langchain-openai`、`langgraph`，但**缺 `langchain` 本体**
2. `main.py` 第 9 行 `import langchain` 导致启动即崩
3. 本地 venv 装了 `langchain==1.3.11` 所以能跑，Docker 只装 requirements.txt 的包
4. `.dockerignore` 排除 `.env`，`backend/.env` 不会进镜像；Docker 只读根目录 `.env`
5. `deploy.sh` 生成根目录 `.env` 时未写 LLM 三件套

### 修复（3 个文件）
1. `requirements.txt`：添加 `langchain>=1.3.0`
2. `.env.example`：添加 `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL` 模板
3. `deploy.sh`：交互式收集 LLM 配置并写入根目录 `.env`

### 服务器紧急修复步骤
1. 编辑 `.env` 补充 LLM 三行
2. `docker compose build backend && docker compose up -d backend`

## 生产环境 WebSocket 连接失败修复

### 问题
- 部署到云服务器后，前端访问 `http://106.53.59.88/chat` 时 WebSocket 连接报错：
  `WebSocket connection to 'ws://localhost:8000/api/ws/chat/...' failed: net::ERR_BLOCKED_BY_LOCAL_NETWORK_ACCESS_CHECKS`
- 聊天界面反复显示「连接中断，请检查网络后重试」

### 根因
- `frontend/src/api/ws.js` 第 6 行：`const WS_BASE = import.meta.env.VITE_WS_URL || "ws://localhost:8000"`
- 生产环境 `VITE_WS_URL` 未设置时，fallback 到 `ws://localhost:8000`
- 浏览器从公网页面访问 `localhost:8000` 被 Local Network Access 安全策略拦截
- 对比 `client.js`（HTTP API）已有生产环境 fallback 到同源路径，但 `ws.js` 没有

### 修复（2 个文件）
1. `frontend/src/api/ws.js`：生产环境 fallback 改为空字符串，走同源 `/api/ws/...` 路径
2. `frontend/.env.production`：补充 `VITE_WS_URL=` 模板

### 服务器操作步骤
1. `git pull`
2. `docker compose up -d --build frontend`（前端容器重新构建，JS 才会使用新的 WS_BASE）

## 生产环境 WebSocket 403 修复

### 问题
- WebSocket URL 已修正为 `ws://106.53.59.88/api/ws/chat/...` 后，握手阶段返回 **403**
- 错误：`Error during WebSocket handshake: Unexpected response code: 403`

### 根因
- `backend/main.py` 中 `CORSMiddleware` 的 `allow_origins` 只配置了 localhost 来源
- 浏览器从 `http://106.53.59.88` 发起 WebSocket 握手时，FastAPI 校验 Origin 不匹配，返回 403
- HTTP API 请求走 Nginx 同源代理，不触发 CORS；WebSocket 握手浏览器强制发送 Origin，会被 CORSMiddleware 拦截

### 修复（4 个文件）
1. `backend/config/settings.py`：新增 `CORS_ORIGINS` 从环境变量读取
2. `backend/main.py`：`allow_origins` 改用 `CORS_ORIGINS` 配置
3. `.env.example`：增加 `CORS_ORIGINS` 模板
4. `deploy.sh`：部署时交互式收集 `CORS_ORIGINS`；JWT_SECRET 改为写入根目录 `.env`（Docker 实际读取的 env_file）

### 服务器操作步骤
1. 本地 `git push` 后，服务器 `git pull`
2. 编辑 `.env` 添加：`CORS_ORIGINS=http://106.53.59.88`
3. `docker compose up -d --build backend`（重新构建，CORS 配置才会生效）

 # AI Agent Assistant 项目面试准备指南
 
 ## 一、项目整体架构
 
 **简介**：基于 FastAPI + LangGraph + React 构建的企业级 AI Agent 平台，集成 RAG 知识库、Tool Calling、Dashboard 监控、多租户等能力。
 
 **技术栈**：FastAPI | LangGraph | React 19 | MySQL | FAISS / pgvector | SQLAlchemy | DashScope API | JWT | SSE / WebSocket | Docker Compose + Nginx | Vite 8
 
 **核心架构**：
 - **后端**：FastAPI 异步框架，4 种聊天通道（同步/SSE流式/WebSocket/直接图调用）
 - **Agent 引擎**：双层设计 — FastRouter（零LLM规则路由） + LangGraph 循环图（Supervisor → 3 Workers → Synthesizer）
 - **RAG**：文档加载（7种格式）→ 文本切片 → DashScope Embedding → FAISS/pgvector 向量检索
 - **工具系统**：ToolManager 注册中心 + ToolRouter（规则/LLM/兜底三级路由）
 - **多租户**：JWT 认证 + tenant_id 字段隔离
 - **前端**：React 19 + Vite 8 + React Router 7，8个页面（Login/Dashboard/Chat/Knowledge/Tools/Logs/Settings/Profile）
 - **部署**：Docker Compose（MySQL + Backend + Frontend/Nginx） + pgvector 可选
 
 ----
 
 ## 二、Agent 工作流架构图
 
 ```mermaid
 flowchart TD
     USER[用户输入] --> FastRouter
    
     subgraph L1 [Layer 1: FastRouter]
         FR[FastRouter<br/>正则/关键词规则匹配<br/>零LLM调用 ~1ms] -->|命中| Direct[直接执行工具<br/>Weather/Calculator/<br/>Greeting/DateTime]
         FR -->|未命中| L2
     end
    
     subgraph L2 [Layer 2: LangGraph 循环图]
         Supervisor[Supervisor 节点<br/>LLM 路由 + 启发式兜底] -->|research| R[Research Worker<br/>知识库检索 Agent]
         Supervisor -->|data| D[Data Worker<br/>SQL 数据分析 Agent]
         Supervisor -->|general| G[General Worker<br/>通用问答 Agent]
         Supervisor -->|FINISH| Check{需 Synthesize?}
        
         R --> Loop{steps < 6?}
         D --> Loop
         G --> Loop
         Loop -->|是| Supervisor
         Loop -->|否| Check
         Check -->|1个 Worker| Final[最终回答]
         Check -->|>=2 个 Worker| Syn[Synthesize Node<br/>聚合多 Worker 结果]
         Syn --> Final
     end
    
     Direct --> UserOut[返回用户]
     Final --> UserOut
 ```
 
 ### 关键设计决策
 
 1. **为什么是双层架构？**
    - 简单请求（天气、计算、问候）无需 LLM 推理，FastRouter 规则匹配即可 ~1ms 返回
    - 复杂请求（多步推理、知识检索+数据分析）走 LangGraph 循环图，Supervisor 智能调度
    - 兼顾快速响应和复杂能力
 
 2. **为什么是循环图而不是链式图？**
    - Worker 执行完毕后不确定是否要换 Agent，所以回到 Supervisor 重新判断
    - 支持"先查知识库 → 再从数据库取数 → 综合分析"的多步任务
    - 安全阀：最多 6 步，避免死循环
 
 3. **为什么 Worker 完成后需要 Synthesize？**
    - 单个 Worker 直接返回
    - 多个 Worker（如 research + data）需 Synthesizer 聚合去重、整合后给用户统一回答

 ## 三、核心模块详解
 
 ### 1. FastRouter（快速规则路由）
 
 **位置**：所有请求的入口，优先在 LangGraph 之前执行
 **原理**：正则表达式 + 关键词匹配，无需任何 LLM 调用
 **匹配规则**（按优先级）：
 - greeting → GreetingTool（问候语）
 - datetime → DateTimeTool（当前时间）
 - calculator → CalculatorTool（数学表达式如 "123+456"）
 - weather → WeatherTool（如 "北京天气"）
 - exchange_rate → HttpTool（如 "美元汇率"）
 
 **匹配成功**：直接执行工具 → Formatter（转自然语言）→ 返回用户，全程 ~1ms
 **匹配失败**：降级到 Layer 2 LangGraph 循环图
 
 **返回值**：FastRouteMatch 对象（tool_name, tool_args, rule_name, is_final）
 
 ### 2. LangGraph 循环图
 
 **核心节点**：
 
 | 节点 | 职责 | 技术实现 |
 |------|------|----------|
 | **Supervisor** | 分析用户意图，决定下一步路由 | LLM 调用 + 启发式关键词兜底 |
 | **Research Worker** | 知识库检索/文档问答 | `create_react_agent` + RAG 工具 |
 | **Data Worker** | SQL 查询/数据分析 | `create_react_agent` + MySQL 工具 |
 | **General Worker** | 通用问答/外部 API | `create_react_agent` + 天气/HTTP 工具 |
 | **Synthesize** | 多 Worker 结果聚合 | LLM 汇总多个来源的回答 |
 
 **Agent State**（循环图的状态）：
 ```
 AgentState {
     messages:       list[BaseMessage]  ← LangChain 消息（Human + AI）
     tenant_id:      int                ← 多租户隔离
     user_id:        int
     next_agent:     str                ← Supervisor 的决策结果
     route_history:  list[str]          ← 已执行过的 Agent 轨迹
     step_count:     int                ← 当前步数（上限 6）
     last_worker:    str                ← 上一个 Worker，防重复路由
 }
 ```
 
 **路由逻辑**（route_from_supervisor）：
 - `FINISH` 且 history 为空 → 默认走 general
 - `FINISH` 且仅有 1 个 Worker → 直接 `end`
 - `FINISH` 且 >= 2 个 Worker → 先走 synthesize
 
 **循环逻辑**（route_after_worker）：
 - step_count < 6 → 回到 supervisor 继续调度
 - step_count >= 6 → 去 synthesize
 
 **Supervisor 启发式兜底**：
 - 关键词评分系统：research（知识库/文档/制度）+ data（SQL/数据库/统计）+ general（天气/你好/计算）
 - 已执行过的 Agent 降权（-2分），避免重复路由
 - 零 LLM 情况下也能做出合理路由决策
 
 ### 3. Tool Calling 框架
 
 **三层架构**：
 ```
 ToolManager（注册中心） → ToolRouter（路由决策） → BaseTool（执行器）
 ```
 
 **ToolManager**（单例模式）：
 - `register()` / `unregister()` / `get()` / `list_tools()`
 - `execute()`（同步）/ `aexecute()`（异步）
 - `get_function_schemas()` → OpenAI function calling 格式
 - 注册 7 个默认工具：Weather / MySQL / Http / RAG / Calculator / DateTime / Greeting
 
 **ToolRouter**（三级路由）：
 1. 关键词规则匹配（零LLM，~1ms）
 2. LLM function calling（规则未命中时，~2-4s）
 3. 关键词兜底（LLM 失败后的最后保障）
 
 **BaseTool**（抽象基类）：
 - 必实现：`name` / `description` / `parameters` / `execute()`
 - `to_function_schema()` → 转为 OpenAI function calling schema
 - `ToolResult` 统一返回结构（success / data / error / execution_time_ms）
 
 ### 4. RAG 知识库
 
 **完整链路**：
 ```
 文档上传 → load_document() → split_text() → embed_texts() → add_vectors() → search()
 ```
 
 **文档加载器**（支持 7 种格式）：
 - PDF → pypdf
 - Word → python-docx
 - Excel → openpyxl
 - PowerPoint → python-pptx
 - TXT/MD → 原生读取
 - 图片 → RapidOCR
 
 **文本切片**：500 字符/块，50 字符重叠
 **Embedding**：DashScope text-embedding-v3（1024 维）
 **向量存储**（双后端）：
 - FAISS（默认）：本地文件索引，内存缓存，`faiss.IndexFlatIP`（内积距离+L2归一化=余弦相似度）
 - pgvector（可选）：PostgreSQL 扩展，`<=>` 余弦距离算子
 
 **检索**：用户问题 → Embedding → FAISS/pgvector 搜索 → 返回 top-5 文档片段 + 相关度评分
 
 ### 5. 流式对话（SSE + WebSocket）
 
 **双通道设计**：
 - **SSE**（`/api/chat/stream`）：HTTP 长连接，逐 token 推送，轻量级
 - **WebSocket**（`/api/ws/chat/{conv_id}`）：全双工，支持 Agent 路由轨迹推送 + 工具状态推送
 
 **SSE 事件格式**：
 - `data: {"type": "chunk", "content": "..."}` → 逐 token 流式输出
 - `data: {"type": "done", "content": "...", "conversation_id": 1}` → 完成
 - `data: {"type": "error", "content": "..."}` → 错误
 
 **WebSocket 事件格式**：
 - `{"type": "token", "content": "..."}` → 流式 token
 - `{"type": "route", "agent": "research", "label": "知识检索", "trail": [...]}` → Agent 路由切换通知
 - `{"type": "tool_start", "tool": "...", "input": {...}}` → 开始调用工具
 - `{"type": "tool_end", "tool": "...", "output": "..."}` → 工具调用完成
 - `{"type": "done", "content": "...", "conversation_id": 1, "route_trail": [...]}` → 完成
 - `{"type": "ping"}` → 心跳（每 30s）
 
 **状态管理**：ChatContext（React Context 全局管理），页面切换不中断 AI 回复
 
 ### 6. 多租户 + JWT 认证
 
 **模型字段**：User.tenant_id, Conversation.tenant_id
 **认证流程**：
 ```
 登录 → JWT 签发（HS256，7天有效期） → Bearer Token 注入 →
 get_current_user（解析token + 查DB） → require_tenant_access（验证租户）
 ```
 
 **角色权限**：
 - `admin`：全部权限
 - `member`：基础使用权限
 - `require_permission(role)` 依赖注入控制接口访问
 
 ### 7. 前端架构
 
 **路由**（React Router 7，`ProtectedRoute` 包装认证）：
 - `/login` → LoginPage
 - `/dashboard` → DashboardPage（默认首页）
 - `/chat` → ChatPage（聊天 + 快捷操作）
 - `/knowledge` → KnowledgePage（知识库管理）
 - `/tools` → ToolCenterPage（工具列表 & 测试）
 - `/logs` → LogsPage（操作日志）
 - `/settings` → SettingsPage
 - `/profile` → ProfilePage
 
 **状态管理**：ChatContext（全局 WebSocket 聊天状态）
 **HTTP 客户端**：Axios 实例 + 自动 token 注入 + 401 跳转
 **UI**：无 UI 框架，自定义 CSS
 
 ### 8. 部署
 
 **Docker Compose 服务**：
 - `db`：MySQL 8.0（3307:3306）
 - `backend`：FastAPI（8000）
 - `pgvector-db`：PostgreSQL + pgvector（5433:5432，可选）
 - `frontend`：Nginx 静态服务（80:80）
 
 **Nginx**：反向代理 /api 到 backend，静态文件由前端容器直接提供

 ## 四、高频面试题 Q&A
 
 ### Q1：这个项目的核心难点是什么？你是如何解决的？
 
 **难点 1：Agent 工作流设计** — 如何兼顾简单请求的快速响应和复杂任务的多步推理？
 **解决**：双层架构。FastRouter 零 LLM 规则匹配（~1ms）覆盖天气/计算/问候等简单请求；LangGraph 循环图（Supervisor + 3 Workers + Synthesize）处理需要多步推理的复杂任务。Supervisor 用 LLM 做路由决策 + 启发式关键词兜底保障可靠性。
 
 **难点 2：流式输出的实时性和交互性** — 用户需要看到 AI 逐字输出，同时了解 Agent 的思考过程。
 **解决**：SSE + WebSocket 双通道。WebSocket 全双工设计不仅推送 token，还推送 Agent 路由轨迹（route/trail）和工具调用状态（tool_start/tool_end），让用户知道 AI 当前的执行阶段。
 
 **难点 3：多租户隔离下的数据安全** — 不同企业的知识库和对话数据不能互相访问。
 **解决**：JWT 认证 + tenant_id 字段隔离 + `require_tenant_access` 依赖注入装饰器。FastAPI 的依赖注入系统统一在每个接口入口检查用户租户权限。
 
 ### Q2：LangGraph 为什么选择循环图而不是 DAG 拓扑？
 
 Worker 执行完后 Supervisor 不确定下一步是结束、继续同类型 Worker、还是换另一种 Worker。循环图的核心优势：
 1. 支持"查资料 → 查数据库 → 综合分析"的多步复杂任务
 2. Worker 执行结果影响下一次路由决策（例如 research 查到数据后不需要 data worker）
 3. 安全性：MAX_GRAPH_STEPS=6 + 重复路由检测 + 空 history 默认走 general
 
 ### Q3：RAG 的架构链路是怎样的？为什么选择 FAISS 而不是直接用 LLM？
 
 文档上传 → 多格式解析（PDF/Word/Excel/PPT/TXT/图片OCR） → 文本切片（500字符，50重叠） → DashScope Embedding（1024维） → 存入 FAISS IndexFlatIP → 检索时问句Embedding → FAISS 相似度搜索 → 返回 top-5 片段。
 
 选 FAISS 的原因：
 - 本地文件索引，无需外部服务，部署简单
 - 内存缓存索引和文档，检索延迟低（~10ms）
 - IndexFlatIP + L2归一化 = 余弦相似度，语义匹配效果好
 - 可选 pgvector 替代，支持更高并发
 
 ### Q4：Streaming Response 是如何实现的？对比 SSE 和 WebSocket？
 
 | 特性 | SSE | WebSocket |
 |------|-----|-----------|
 | 传输层 | HTTP 长连接 | TCP 全双工 |
 | 消息方向 | 服务器→客户端 | 双向 |
 | 应用场景 | 简单的文本流式输出 | 带路由/工具状态的复杂交互 |
 | 心跳 | 不需要 | 需要（每30s） |
 | 重连 | 浏览器原生支持 | 手动实现 |
 | 复杂度 | 低 | 高 |
 
 项目同时实现两种，WebSocket 是主力方案，SSE 作为轻量备选。
 
 ### Q5：Tool Calling 如何实现"规则优先"？给你一个问句，能讲清楚完整的路由链路吗？
 
 **用户输入"北京天气"**：
 1. FastRouter 正则匹配 → 命中 weather 规则 → 提取 city="北京", days=1 → 直接调用 WeatherTool
 2. ToolManager.execute("weather", city="北京", days=1) → 调用高德天气 API
 3. Formatter.format_tool_result() → 转为"📍 北京\n🌤 天气：晴\n🌡 温度：25°C..."
 4. 返回用户，全程 ~1s（无 LLM 调用）
 
 **用户输入"分析上个月的销售数据"**：
 1. FastRouter 未命中（无匹配规则）
 2. 进入 LangGraph Supervisor → LLM 路由 → 判定为 data
 3. Data Worker (create_react_agent) → 调用 MySQLTool → 执行 SQL
 4. Worker 完成后回到 Supervisor → 判定 FINISH
 5. route_history=["data"] 仅 1 个 Worker，无需 synthesize → 直接返回
 
 **用户输入"查一下报销制度，然后算算我上个月的报销总额"**：
 1. FastRouter 未命中
 2. Supervisor → research（先查知识库了解报销制度）
 3. Research Worker → RAGTool → 检索 FAISS → 返回报销制度文档
 4. Worker → 回到 Supervisor → 判断还需要 data
 5. Supervisor → data → Data Worker → MySQLTool → 查询数据库
 6. Worker → 回到 Supervisor → step_count=3 → FINISH
 7. route_history=["research","data"] → 2个 Worker → 走 Synthesize
 8. Synthesize Node 聚合两次结果 → 返回综合回答
 
 ### Q6：项目中哪些地方用了 AI，哪些没有用 AI？
 
 **用 AI 的地方**：
 - LangGraph Supervisor 的路由决策
 - Research/Data/General Worker 的对话生成
 - Synthesize 节点的结果聚合
 - ToolRouter 的 LLM function calling（兜底阶段）
 - Embedding 模型（文本向量化）
 
 **没有用 AI 的地方**（纯规则/算法）：
 - FastRouter 规则匹配（正则表达式）
 - Supervisor 启发式兜底（关键词评分）
 - ToolRouter 关键词规则匹配
 - FAISS 向量相似度搜索（余弦相似度算法）
 - 文本切片（固定大小）
 - JWT 认证解码
 - 数据库 CRUD
 
 ### Q7：Dashboard 监控了哪些数据？怎么实现的？
 
 **监控指标**：
 - 今日调用次数（用户当日消息数）
 - 知识库文档总数
 - 近7天消息趋势
 - Token 消耗统计
 - 系统状态（运行时间、模型信息、向量库数量）
 
 **实现**：`crud/dashboard.py` 用 SQLAlchemy 聚合查询（COUNT + JOIN + GROUP BY + CASE WHEN）+ `router/dashboard.py` FastAPI 路由 + 前端用柱状图展示
 
 ### Q8：如何保证系统的可扩展性？
 
 1. **新工具**：继承 BaseTool → 实现 name/description/parameters/execute() → `ToolManager.register()` → 自动出现在 LangChain tool schema 中
 2. **新 Worker Agent**：LangGraph 的 state 和路由是插件化的，只需新增 node + 路由分支
 3. **新向量库后端**：实现 add_vectors / search / count / remove_by_source 接口即可替换 FAISS
 4. **新文档格式**：在 `rag/loader/` 中加一个 load_xxx() 函数
 5. **多租户**：tenant_id 字段天然支持水平扩展
 
 ### Q9：FastAPI 在这个项目中用了哪些特性？
 
 - **异步支持**：`async def` + `await` 处理聊天、Embedding、SSE 流式响应
 - **依赖注入**：`Depends(get_current_user)` 统一管理 JWT 认证、租户权限验证
 - **Pydantic 模型**：`ChatRequest`, `ChatResponse` 请求/响应校验
 - **WebSocket**：`/api/ws/chat/{id}` 全双工通信
 - **StreamingResponse**：SSE 流式输出，逐 token 推送
 - **异常处理**：全局 `exception_handlers`（HTTPException / IntegrityError / SQLAlchemyError / 通用）
 - **lifespan**：启动时建表 + 迁移旧表 + 注册默认工具
 - **中间件**：CORS 跨域配置
 
 ### Q10：项目中如何保证聊天历史的连续性？
 
 1. **数据库存储**：MySQL 的 `conversations` + `messages` 表存储完整对话
 2. **历史加载**：每次新请求从 DB 加载最近 20 条消息，转为 LangChain BaseMessage 格式传入 AgentState
 3. **消息转换**：`db → HumanMessage(content=...)` / `AIMessage(content=...)`
 4. **上下文管理**：Supervisor 的 LLM 调用传入 `state["messages"][-8:]` 最近 8 条消息
 5. **标题自动更新**：第一条用户消息自动设为对话标题的前 30 字
 
 ### Q11：Docker 部署的流程是怎样的？
 
 `docker-compose up -d --build`：
 1. **db**：MySQL 8.0 启动，healthcheck 等待就绪
 2. **backend**：Dockerfile 安装 Python 依赖 → FastAPI 启动（lifespan 中建表+注册工具）
 3. **frontend**：Dockerfile 构建 React → Nginx 服务静态文件 + 反向代理 /api → backend
 4. **pgvector-db**：可选，通过 profile 控制（`--profile pgvector`）
 5. 卷挂载：uploads（文件） / rag/data（向量索引） / mysql_data（数据库）
 
 ### Q12：WebSocket 的心跳机制和重连是怎么做的？
 
 **心跳**：服务端每 30s 发送 `{"type": "ping"}`，`asyncio.create_task(_heartbeat())` 异步任务管理
 **取消**：WebSocket 断开时 `heartbeat_task.cancel()`
 **重连**：前端 `ws.js` 中 `connectChatWs()` 每次发消息时新建连接，`sendChatMessage()` 检测 `readyState === CONNECTING` 时用 `addEventListener('open')` 等待连接就绪
 
 ### Q13：LangGraph 的 Agent State 为什么用 TypedDict？messages 为什么用 add_messages 归约？
 
 `TypedDict` 是 LangGraph 支持的类型化状态定义，比 `dict` 更安全，IDE 有自动补全。`add_messages` 归约器会自动合并多个节点输出的消息列表，避免手动拼接，而且保证消息顺序不乱。
 
 ### Q14：项目中 Auth 模块的完整链路是什么？
 
 登录 `/api/users/login` → 验证用户名密码（bcrypt） → 生成 JWT（HS256, `sub`=user_id, `exp`=7天） → 前端存 localStorage
 后续请求 → Axios interceptor 自动注入 `Authorization: Bearer <token>` → FastAPI `Depends(get_current_user)` 解析验证 → `require_tenant_access()` 检查租户
 401 响应 → Axios interceptor 清除 token → 跳转 `/login`
 
 ### Q15：FAISS 和 pgvector 如何做选择？
 
 | 场景 | 推荐 |
 |------|------|
 | 单机低并发 | FAISS |
 | 多服务实例共享索引 | pgvector |
 | 快速原型 | FAISS |
 | 生产环境，需要持久化 | pgvector |
 | 不需要额外服务 | FAISS |
 | 已有 PostgreSQL 集群 | pgvector |
 
 项目默认 FAISS，通过 `VECTOR_STORE_PROVIDER=pgvector` 一键切换。

 ## 五、Quick Reference
 
 ```bash
 # 本地启动
 cd backend && python main.py
 cd frontend && npm run dev
 
 # Docker 部署
 docker-compose up -d --build
 
 # 查看日志
 docker-compose logs -f backend
 docker-compose logs -f frontend
 
 # 环境变量
 cp .env.example .env
 # 必须配置：DATABASE_URL, JWT_SECRET, DASHSCOPE_API_KEY
 
 # 技术测试
 curl http://localhost:8000/health
 curl http://localhost:8000/db-test
 ```

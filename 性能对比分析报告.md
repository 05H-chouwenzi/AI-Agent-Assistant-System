# AI 助手性能对比分析报告

> 对比对象: 你的项目 vs WL7749/ai-agent（参考项目，1秒内响应）

---

## 一、两个项目架构共同点

| 层面 | 你的项目 | 参考项目 |
|------|---------|---------|
| 框架 | FastAPI + LangGraph | FastAPI + LangGraph |
| LLM | DeepSeek (DashScope 兼容) | DeepSeek (直连) |
| Agent 编排 | Supervisor → Worker(create_react_agent) | Supervisor → Worker(create_react_agent) |
| 路由 | 启发式关键词 + FastRouter | LLM 路由 + 启发式关键词兜底 |
| 向量库 | FAISS (1024维) | pgvector (1536维) |
| 流式方式 | SSE + WebSocket | WebSocket |

**结论：架构高度相似，但实现细节天差地别。**

## 二、性能差距的根本原因

### 问题一：【致命】WebSocket 是"假流式"（ws_chat.py:104）

```
你的 ws_chat.py：                     参考 ws.py：
ainvoke(state)                       astream_events(state, v2)
  └─ 等待 DeepSeek 完整返回               └─ 实时监听 on_chat_model_stream
     (3-8s)                                   └─ 逐 token 推送到前端
       └─ 拿到完整回答                            (首个 token 500ms-1s)
          └─ 每 4 字符 + asyncio.sleep(0.01) → 伪流式
```

**影响：用户看到第一个 token 的等待时间 = 完整 LLM 响应时间 (3-8秒)**

### 问题二：【严重】SSE 流式有"双重执行"Bug（chat_stream.py:100-105）

```python
# astream_events 可能在某些情况下不产生 token
async for event in agent_graph.astream_events(state, version="v2"):
    ...  # 如果这里没收到任何 token
    # ↑ supervisor 的 token 被过滤掉了（第94行）
    # ↑ 如果 worker 也没产生 token（异常/timeout）

# fallback: 再来一次完整的 ainvoke！
if not full_answer:
    result = await agent_graph.ainvoke(state)  # ← 第二次 LLM 调用！
```

**影响：最坏情况下一次请求 = 两次完整 LLM 调用 (6-16秒)**

### 问题三：【严重】同步阻塞工具炸毁事件循环

| 工具 | 文件:行号 | 阻塞方式 | 每次延迟 |
|------|----------|---------|---------|
| HTTP 请求 | `tools/http_tool.py:128` | `urllib.request.urlopen()` 同步 HTTP | 最多 15s |
| 天气查询 | `tools/weather_tool.py:84` | `httpx.get()` 同步 HTTP | 最多 3s |
| MySQL 查询 | `tools/mysql_tool.py:109` | 新建 `pymysql.connect()` 无连接池 | 50-200ms |
| 向量嵌入 | `rag/embedding/__init__.py:14` | `OpenAI.embeddings.create()` 同步 | 100-500ms |
| LLM 路由 | `tools/tool_router.py:292` | 新建同步 `OpenAI` 客户端 + 同步调用 | 2-5s |

**关键问题：这些同步调用在 `async def` 上下文中执行，阻塞了整个 FastAPI 事件循环，导致所有并发请求排队等待。**

### 问题四：【中等】模型差异

```
你的: qwen3.6-27b (DashScope 代理) → 阿里云，27B 参数，推断速度较慢
参考: deepseek-chat (直连) → DeepSeek 官方，优化的推理速度
```

## 五、修复方案（按优先级）

### P0：WebSocket 真流式化（预期减少 5-7s 首 token 延迟）

```python
# ws_chat.py:104 — 当前
result = await agent_graph.ainvoke(state)  # ❌ 非流式

# 改为
async for event in agent_graph.astream_events(state, version="v2"):
    if kind == "on_chat_model_stream":
        if node != "supervisor":  # 过滤 supervisor token
            await websocket.send_json({"type": "token", "content": chunk.content})
```

**操作：复制 chat_stream.py 的 astream_events 逻辑到 ws_chat.py，删除伪流式代码**

### P0：修复 SSE 双重执行 Bug（预期减少 3-5s）

```python
# chat_stream.py:100-105 — 当前
if not full_answer:
    result = await agent_graph.ainvoke(state)  # ❌ 第二次 LLM 调用

# 改为
if not full_answer:
    yield sse_event("error", "AI 未返回有效回答")
    return  # ← 不要再调第二次 LLM
```

### P1：工具异步化（预期减少阻塞延迟 3-15s）

```python
# weather_tool.py — 当前
resp = httpx.get(url, timeout=3.0)  # ❌ 同步

# 改为
async with httpx.AsyncClient(timeout=3.0) as client:
    resp = await client.get(url)  # ✅ 异步
```

```python
# http_tool.py — 当前
urllib.request.urlopen(req, timeout=15)  # ❌ 同步 urllib

# 改为
async with httpx.AsyncClient(timeout=15.0) as client:
    resp = await client.get(url)  # ✅ 异步
```

```python
# mysql_tool.py — 当前
conn = pymysql.connect(...)  # ❌ 每次新建连接

# 改为：复用 SQLAlchemy 连接池
from database.session import engine
with engine.connect() as conn:  # ✅ 连接池复用
```

```python
# embedding/__init__.py — 当前
from openai import OpenAI           # ❌ 同步客户端
resp = client.embeddings.create(...)

# 改为
from openai import AsyncOpenAI      # ✅ 异步客户端
resp = await client.embeddings.create(...)
```

### P2：消除 ToolRouter 的重复 LLM 调用

ToolRouter 的 `_llm_route()` 会在 LangGraph 之前又做一次 LLM 调用来"预判"用什么工具。而 LangGraph 的 create_react_agent 本身就带有 function calling。**这是冗余的。**

修改 `tool_router.py`：
- 如果 Agent Graph 已经处理，ToolRouter 只做关键词匹配（不改 LLM 调用）
- 或者：直接删除 `_llm_route` 方法，全部走 FastRouter 关键词 + Agent Graph

### P2：direct_llm.py 接入主流程（预期更简洁的架构）

项目里已经有 `services/direct_llm.py`，对标参考项目的极简架构：
- 单一 LLM 调用（无多 Agent 路由）
- 真异步流式 (AsyncOpenAI)
- 但使用的是 DashScope 配置而非 LLM 配置

**修复 direct_llm.py 的配置**，使其使用正确的 LLM_API_KEY/LLM_BASE_URL，然后作为替代方案。

## 六、总结：为什么参考项目能做到 1 秒

| 参考项目做了什么 | 你的项目没做什么 |
|-----------------|----------------|
| ✅ `astream_events` 真流式 | ❌ WebSocket 用 `ainvoke` 假流式 |
| ✅ 所有工具都用 `@tool async def` | ❌ 工具全是同步阻塞 |
| ✅ `httpx.AsyncClient` 异步 HTTP | ❌ `urllib` + 同步 `httpx` |
| ✅ PostgreSQL + asyncpg 异步DB | ❌ MySQL 工具用 `pymysql` 同步 |
| ✅ 无多余的 ToolRouter LLM 调用 | ❌ ToolRouter 又额外调 LLM |
| ✅ Embedding 用 hash(伪,但快) | ❌ Embedding 同步调 DashScope API |
| ✅ 无双重执行 Bug | ❌ SSE fallback ainvoke |

**核心矛盾：你的项目在每个环节都有"一点点慢"，加起来就从 1 秒变成了 10 秒。**

## 七、建议实施顺序

1. **先修 WebSocket 真流式**（改动最小，效果最大：-5~7s 首 token 延迟）
2. **再修 SSE 双重执行 Bug**（一行改动，避免故障时翻倍延迟）
3. **工具异步化**（4 个工具逐个改，测试后合入）
4. **消除 ToolRouter 额外 LLM 调用**（简化逻辑，减少延迟）
5. **Embedding 缓存或异步化**（减少 RAG 场景延迟）
6. **考虑 direct_llm.py 作为替代方案**（更简单的架构，对标参考项目）

# AI 助手响应延迟分析报告

> 对比项目: [ai-qa-community](https://github.com/WL7749/ai-qa-community) (~1秒响应) vs 本项目 (~10秒响应)

---

## 核心结论：架构复杂度是根本原因

参考项目之所以能在 1 秒内响应，是因为它走的是**最短路径**：

```
用户提问 → DeepSeek API (stream=True) → SSE 逐 token 推送 → 存数据库
```

**只有 1 次 LLM 调用，且是真正的流式输出**，用户几乎立刻看到第一个 token。

而你的项目走的是：

```
用户提问 → JWT鉴权 → FastRouter → 创建会话/加载历史 → Supervisor路由 → create_react_agent
→ (可能调工具) → LLM推理 → LLM解读 → BUG:双重执行 → FAISS磁盘读 → 同步阻塞 → SSE推送
```

**每条请求至少 1-3 次 LLM 调用，且存在严重 BUG 导致翻倍**。

## P0 级问题（必须立刻修复）

### 1. chat_stream.py 双重图执行 BUG

**文件**: `backend/api/chat_stream.py` 第 99-104 行

```python
# ❌ BUG: 这段代码在 async for 循环内部！
async for event in agent_graph.astream_events(state, version="v2"):
    ...
    if kind == "on_chat_model_stream":
        ...
        full_answer += chunk.content
        yield sse_event("chunk", chunk.content)

    # ⚠️ 缩进与 if kind == "on_chat_model_stream" 同级 → 在循环内
    if not full_answer:                              # ← 每个非 token 事件都触发
        result = await agent_graph.ainvoke(state)    # ← 整图重跑一次！
        ...
```

**影响**:
- `astream_events` 先发出多个非 token 事件（`on_chain_start` 等），每次都触发 `ainvoke`
- 导致整个 LangGraph 图被执行 **2 次甚至多次**
- LLM 调用次数翻倍 → 响应时间翻倍
- `ainvoke` 设置 `full_answer` 后，后续 `astream_events` 的 token 事件继续追加 → 用户看到**重复内容**

**修复**:

```python
# ✅ 正确: 将 fallback 移到循环外
async for event in agent_graph.astream_events(state, version="v2"):
    kind = event.get("event")
    metadata = event.get("metadata")
    node = metadata.get("langgraph_node", "") if isinstance(metadata, dict) else ""

    if kind == "on_chat_model_stream":
        data = event.get("data")
        chunk = data.get("chunk") if isinstance(data, dict) else None
        if chunk and hasattr(chunk, "content") and chunk.content:
            if node == "supervisor":
                continue
            full_answer += chunk.content
            yield sse_event("chunk", chunk.content)

# 循环结束后，如果没收到 token → fallback
if not full_answer:
    result = await agent_graph.ainvoke(state)
    last = result["messages"][-1]
    if isinstance(last, AIMessage):
        full_answer = last.content if isinstance(last.content, str) else str(last.content)
        yield sse_event("chunk", full_answer)
```

### 2. Worker Agent 使用非流式 LLM → SSE 流式名存实亡

**文件**: `backend/agent/graph/nodes.py` 第 43 行

```python
llm = get_llm(streaming=False)  # ← 非流式！
```

**影响**:
- `create_react_agent` 内部 LLM 调用是非流式的 → `ainvoke` 等待完整响应才返回
- `chat_stream.py` 用 `astream_events` 捕获 `on_chat_model_stream`，但因为 LLM 非流式，这些事件**不触发或只触发一次完整块**
- 用户在 SSE 端看到的不是逐 token 流式输出，而是**等完整响应后才收到**
- 这也是问题 #1 的 fallback 逻辑被触发的原因——因为流式事件经常不产出 token

**修复选项**:

**方案 A（推荐）**: 改用流式 LLM，让 SSE 真正逐 token 推送

```python
llm = get_llm(streaming=True)  # 改为流式
```

同时需要给 `get_llm(streaming=True)` 加缓存：

```python
_llm_streaming_instance: ChatOpenAI | None = None

def get_llm(*, streaming: bool = True) -> ChatOpenAI:
    global _llm_instance, _llm_streaming_instance
    if streaming:
        if _llm_streaming_instance is None:
            _llm_streaming_instance = ChatOpenAI(
                model=LLM_MODEL, api_key=LLM_API_KEY, base_url=LLM_BASE_URL,
                streaming=True, temperature=0.3,
            )
        return _llm_streaming_instance
    if _llm_instance is None:
        _llm_instance = ChatOpenAI(...)
    return _llm_instance
```

**方案 B（架构级重构）**: 参考 ai-qa-community 的极简架构，不走 LangGraph agent 路径

## P1 级问题（显著影响性能）

### 3. FAISS 索引每次搜索都从磁盘重新加载

**文件**: `backend/rag/vector_store/faiss_store.py`

```python
def _get_index() -> faiss.IndexFlatIP:
    return faiss.read_index(str(index_file))  # ← 每次搜索都读磁盘

def _load_docs() -> list[dict]:
    return pickle.load(f)  # ← 每次搜索都反序列化
```

**影响**: 每次 RAG 检索至少 2 次磁盘 I/O + pickle 反序列化，100-500ms

**修复**: 内存缓存索引和文档，启动时加载

```python
_index_cache: faiss.IndexFlatIP | None = None
_docs_cache: list[dict] | None = None

def _get_index() -> faiss.IndexFlatIP:
    global _index_cache
    if _index_cache is None:
        index_file = _STORE_DIR / "index.faiss"
        if index_file.exists():
            _index_cache = faiss.read_index(str(index_file))
        else:
            _index_cache = faiss.IndexFlatIP(VECTOR_DIM)
    return _index_cache

def _load_docs() -> list[dict]:
    global _docs_cache
    if _docs_cache is None:
        doc_file = _STORE_DIR / "documents.pkl"
        if doc_file.exists():
            with open(doc_file, "rb") as f:
                _docs_cache = pickle.load(f)
        else:
            _docs_cache = []
    return _docs_cache
```

注意：`add_vectors` 和 `remove_by_source` 写操作后需要清除缓存（设为 None），下次搜索时重新加载。

## 架构级建议：为什么参考项目这么快？

参考项目 `ai-qa-community` 的核心流式端点：

```python
stream = deepseek_client.chat.completions.create(
    model="deepseek-chat",
    messages=[...],
    stream=True,           # ← 真正的流式
    timeout=120,
)
for chunk in stream:
    delta = chunk.choices[0].delta.content or ""
    if delta:
        yield f"data: {json.dumps({'type': 'token', 'content': delta})}\n\n"
```

**极简架构**:
- **0 层路由决策**：不需要 supervisor/planner/router
- **1 次 LLM 调用**：直接流式调用 DeepSeek
- **真正的 SSE**：每个 token 立刻推送
- **无 Agent 框架开销**：没有 LangGraph、没有 ReAct、没有多轮推理

### 两种优化路径

**路径 1: 修复现有架构（渐进式）**

优先修复 P0 和 P1 问题，预期可将响应时间从 ~10s 降到 ~3-5s：
- 修复双重执行 BUG → 立刻砍掉一半延迟
- Worker 改用流式 LLM → SSE 真正逐 token 推送，用户体验从"等10秒"变成"1秒看到第一个字"
- FAISS 内存缓存 → RAG 检索提速
- 同步改异步 → 消除事件循环阻塞

**路径 2: 简化架构（激进式，参考 ai-qa-community）**

对于"普通对话"场景（不需要工具调用），走极简路径：

```
FastRouter 命中 → 执行工具 (0次LLM, ~1ms)
FastRouter 未命中 → direct_llm.stream_llm (1次LLM, 真正流式)
```

你的项目其实已经有 `services/direct_llm.py` 这个模块了！它实现了和参考项目完全一致的架构：
- `AsyncOpenAI` 异步客户端
- `stream=True` 真正流式
- 单次 LLM 调用

**只是目前没有被 chat_stream.py 使用**。

建议：对于不需要工具的普通对话，直接用 `direct_llm.stream_llm`；只有确实需要工具（RAG、数据库查询等）时才走 LangGraph agent。
